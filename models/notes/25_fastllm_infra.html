1 · 一句话定位 — fastllm 在推理框架 landscape 里的位置

2025-2026 的 LLM 推理框架可以粗分成三条赛道:

赛道代表核心诉求典型用户
高并发 servingvLLM / SGLang / TensorRT-LLMPagedAttention · radix cache · continuous batching · 多机 EP/TP云厂商 / API service provider
本地 / 端侧llama.cpp · MLC-LLM · ExecuTorchGGUF · 端 CPU/Metal/NPU · 最低显存个人开发者 · 手机 · Mac
"超大模型 + 边缘硬件"fastllm · KTransformers · ik_llama.cppMoE expert 卸到 CPU/NUMA · 任意显卡跑 671B · int4 + FP8 混精单机本地部署满血 MoE · 老服务器复用

fastllm 落在第三条赛道。它最接近 KTransformers(清华 KVCache.AI 的"MoE CPU 卸载"路线)和 ik_llama.cpp(llama.cpp 的高性能 fork),但走的是更彻底的 NUMA 优化路线 — README 里明确鸣谢 KTransformers 的"MOE 算子线程不平衡时动态调度的思路",同时基于 numa 改进了自己的 MoE 算子(参考 lvllm 的实现)。

2 · 完整 GPU / 硬件支持矩阵

fastllm 的招牌就是兼容性:README 第一行就写"M40, K80 到 5090 全系列 N 卡"。这是 vLLM / SGLang / TRT-LLM 早就放弃的赛道 — vLLM 从 0.5.0 起逐步抛弃 SM < 70 的卡,而 fastllm 通过 PIP 包仍支持 SM_52 起步(版本日志 V0.0.0.6 提到"PIP 安装包支持更多架构,目前最低支持到 SM_52")。

3 · 5 级硬件资源测试阶梯

把读者从"我电脑只有 2 GB 显存"到"我有 8 张 H100"的全光谱用法整理成五个阶梯:

🟢 阶梯 1 · 入门 (~2 GB+ VRAM, Qwen3-0.6B, 5 分钟)

场景:验证安装、笔记本试水、桌面 1050Ti / Mac CPU(注意 Mac GPU 不支持)。

pip install ftllm -U
ftllm run Qwen/Qwen3-0.6B

说明:0.6B 模型 FP16 约 1.2 GB,在 2 GB 显存的卡上能跑;CPU-only 也行(--device cpu)。Webui 用 ftllm webui Qwen/Qwen3-0.6B,OpenAI 风格 server 用 ftllm server Qwen/Qwen3-0.6B --port 8080。

🟡 阶梯 2 · 单卡小模型 (6-16 GB, Qwen3-8B/14B INT4)

场景:3060/4060 (12 GB) / 2080Ti (11 GB) / 3080 (10 GB) / 桌面工作站。

# 直接在线量化加载 14B → INT4
ftllm server Qwen/Qwen3-14B --dtype int4 --port 8080

# 或者先离线导出再加载(读取更快)
ftllm export /mnt/Qwen3-14B -o /mnt/Qwen3-14B-INT4 --dtype int4 -t 16
ftllm run /mnt/Qwen3-14B-INT4/

14B 模型 INT4 约 7 GB,8B 模型 INT4 约 4 GB,12 GB 显存绰绰有余。
支持的量化类型:int4g(分组量化,精度更好) · int4 · int4g256 · int8 · fp8 · float16 · bfloat16。

🟠 阶梯 3 · 中端 (24 GB+ VRAM, Qwen3-30B-A3B / 32B / GLM4.5-Air)

场景:4090/3090 24 GB · A5000/A6000 · 工作站单卡。

# Qwen3-32B AWQ 量化原版,推荐 24 GB 单卡
ftllm server Qwen/Qwen3-32B-AWQ --device cuda

# Qwen3-30B-A3B MoE 模型,使用单卡 + CPU 混合推理
ftllm server Qwen/Qwen3-30B-A3B --device cuda --moe_device cpu -t 16

# GLM4.5-AIR (支持工具调用)
ftllm server zai-org/GLM-4.5-Air

到这一阶梯就开始体现 fastllm 真正的特长:即使 4090 24 GB 显存不够装下 30B MoE 全部权重,也可以让 attention/non-MoE 留 GPU、MoE expert 卸到 CPU。

🔵 阶梯 4 · 满血 DeepSeek 671B (任意 ≥10 GB GPU + 256-700 GB RAM)

场景:这是 fastllm 的招牌用例 — README 第一行就写。一台带双路 EPYC 9004/9005 服务器(典型配置 512 GB 内存 ~ 768 GB)+ 一张 4090/A100/H100/B200,部署 DeepSeek R1 671B 满血原版 FP8。

# INT4 量化版,~340 GB 内存(MoE)+ ~6 GB 显存(non-MoE)
ftllm server fastllm/DeepSeek-R1-INT4 --device cuda --moe_device numa
# 启动 NUMA 加速
export FASTLLM_USE_NUMA=ON
export FASTLLM_NUMA_THREADS=27   # 每 NUMA 节点 27 线程
# 关闭系统 numa balancing(强烈推荐)
sudo sysctl -w kernel.numa_balancing=0

# FP8 原版(671B FP8 ~ 660 GB,需要 700 GB+ 内存)
ftllm server deepseek-ai/DeepSeek-R1 --dtype fp8 --device cuda --moe_device numa

性能(README 给出):

  • 双路 9004/9005 + 单 GPU 部署 FP8 原版 671B: 单并发 ~20 tps
  • 双路 9004/9005 + 单 GPU 部署 INT4 量化 671B: 单并发 ~30 tps · 多并发 60+ tps

这是单机 671B 部署目前少有的、有数字可量化的方案。

🔴 阶梯 5 · 多卡张量并行 + GPU+CPU 混合张量并行

场景:2-8 张卡的服务器,DeepSeek-V4-Flash / Qwen3-72B 等。fastllm 支持 奇数张卡(3/5/7),这是 vLLM/SGLang TP 罕见的。

# 双卡张量并行
ftllm server fastllm/DeepSeek-V3-0324-INT4 --device multicuda:0,1

# 双卡 + CPU 混合张量并行 (cuda:0 算 4/10, cuda:1 算 5/10, cpu 算 1/10)
ftllm server X --device multicuda:0:4,1:5,cpu:1

# 串行多卡(每张卡负责一段层,而非 TP 切权重)
ftllm server X --device "{'cuda:0':3,'cuda:1':2}"
# 等价简写: --device cudapp=2 (均匀串行)
# 比例串行: --device cudapp=1:2:3

# 高级:大部分 MoE 走 NUMA,少量 MoE 用双卡 TP 加速
ftllm server fastllm/DeepSeek-V4-Flash --device cuda \
  --moe_device "{'multicuda:0,1':15,'numa':85}"

更极致地:"{'cuda':1,'numa':8,'disk':1}" 可以把 1/10 的 MoE 层走磁盘(V0.1.6.2 新增 disk MoE 推理),用 SSD 当容量补充,但速度会比 NUMA 慢得多。

4.2 PyPI 滞后问题

实战经验提示:fastllm 的 PyPI wheel 主要预编译 SM_80 (Ampere) 之后的架构,V0.1.6.0 才把 SM80 加入预编译(版本日志:"CUDA wheel 补充 NCCL 依赖和 SM80 预编译架构")。
对 B200 / 5090 等新架构,首次测试强烈建议源码编译。具体表现:pip install ftllm 安装后启动卡在 no kernel image is available for execution on the device,这就是 CUDA_ARCH 没匹配上,改源码编译指定 -DCUDA_ARCH=100 / 120 即可解决。

5 · 关键概念解构

5.1 CPU + GPU 混合推理(灵魂)

DeepSeek-R1 671B 模型的权重分布大致是:

  • MoE expert 权重(每层 256 个 expert,每 token 激活 8 个): 占总参数 95%+,~650 GB FP8
  • non-MoE 部分(embeddings · attention QKVO · LayerNorm · LM head): 占 ~3-5%,~10-20 GB

但 每次 forward 只激活一小部分 expert(8/256),因此 MoE 部分是带宽 bound,不是计算 bound。这意味着:

attention (compute-heavy, 计算密集) → GPU
MoE expert (memory-bandwidth-heavy, 带宽密集) → CPU + DDR5 多通道

双路 EPYC 9004/9005 服务器有 24 个 DDR5 通道(双 socket × 12 通道),理论内存带宽 ~920 GB/s,接近一张 A100 的 HBM 带宽。fastllm 利用这个特性,把 671B 的 MoE 部分平铺在 RAM,GPU 只算 attention/non-MoE。

5.2 NUMA 加速

双路 / 四路服务器有 2-8 个 NUMA 节点(每个 CPU socket 一个或多个)。如果 OS 自动跨节点搬运数据(numa balancing),性能会严重劣化。fastllm 的 NUMA 优化关键有三点:

  1. 关闭 OS numa balancing: sudo sysctl -w kernel.numa_balancing=0(README 强烈建议)
  2. FASTLLM_USE_NUMA=ON 环境变量激活多 NUMA 模式
  3. FASTLLM_NUMA_THREADS 设定每个 NUMA 节点的线程数(推荐 = socket 物理核数,如 27)

当 --moe_device numa 时,推理线程数主要由 FASTLLM_NUMA_THREADS 决定,-t 参数应设小(推荐 1)。注意 --device numa 和 --moe_device numa 是两个独立参数 — 前者整模型跑 NUMA(用于纯 CPU 推理),后者只有 MoE 层跑 NUMA(混合推理标配)。

如果想单 NUMA 节点跑(避免跨 socket),用 numactl -C 0-31 -m 0 ftllm server X --moe_device cpu -t 27 绑定。

5.3 multicuda 张量并行

fastllm 的 multicuda 是基于自有 worker 池实现的张量并行,支持 3、5、7 等奇数张卡(README 第 5 条亮点功能)。这是 vLLM/SGLang 罕见的特性 — 它们的 TP 通常要求张卡数能整除头数(power-of-2 友好)。

核心环境变量(mixforward.md):

环境变量默认值说明
FASTLLM_DISABLE_MULTICUDA_DEDICATED_WORKERS未设置设 1 关闭专用 worker,回退通用线程池
FASTLLM_MULTICUDA_WORKER_CPU_BASE0专用 worker 绑核起始 CPU,worker i 绑到 (BASE+i) % CPU 数

5.5 --moe_device 完整语法

这是 fastllm 最强的"调参旋钮"。基本格式:

# 单一设备
--moe_device cpu       # 全 MoE 走 CPU
--moe_device numa      # 全 MoE 走多 NUMA 节点
--moe_device cuda      # 全 MoE 走 GPU(显存够时,极少用)

# 多设备按比例(推荐用法,关键!)
--moe_device "{'cuda':1, 'numa':19}"           # 1/20 MoE 上 GPU, 19/20 上 NUMA
--moe_device "{'cuda':1, 'numa':8, 'disk':1}"  # 1/10 GPU + 8/10 NUMA + 1/10 SSD
--moe_device "{'multicuda:0,1':15, 'numa':85}" # 15/100 MoE 走双卡 TP + 85/100 NUMA

这种"按比例分配 MoE 层"是 fastllm 区别于 KTransformers 的关键设计 — 不需要二选一,可以任意切。这也是为什么 fastllm 在"显存有一点剩余想榨干"时往往比 KTransformers 略快(V0.1.6.4 更新日志:"减少 NVFP4 模型的内存占用"也是这条思路的延伸)。

5.6 动态量化(dtype_config)

V0.1.4.8 起 fastllm 支持"通用动态量化" — 用正则表达式指定权重子集的精度。配置文件示例:

[
  {"key": "(.)*",
   "dtype": "int4g",
   "comment": "default int4g"},
  {"key": "model\\.layers\\.([0-9]|[1][0-9]|2[0-5])\\.(.)*",
   "dtype": "fp8",
   "comment": "layer 0~25 use fp8"},
  {"key": "(.)*mlp\\.gate\\.weight",
   "dtype": "float16",
   "comment": "gate use fp16"},
  {"key": "(.)*experts(.)*[gate|up|down]_proj(.)*",
   "dtype": "int4",
   "comment": "moe use int4"}
]

调用:ftllm server X --dtype_config /path/to/config.json。规则越靠后优先级越高。这相当于把 vLLM 的 layer-wise quantization 推到了 weight-name regex 级别 — 比 GGUF 的 Q*_K_L 命名约定更细。

6.3 NUMA / 线程数对性能的影响

FAQ 和 README 都强调:

  • FASTLLM_NUMA_THREADS 一般设为单 socket 物理核数(典型 EPYC 9004 单 socket 32 核 → 27-32)
  • CPU 线程数 不要超过物理核数(hyperthreading 反而拖累)
  • 关闭 kernel.numa_balancing 是性能提升的关键开关
  • 速度和参数设置并不完全正相关(README 第 4 节直说):需要 多方向尝试

6.4 KV cache 与长上下文(V0.1.6.x 新增)

V0.1.6.0 起 fastllm 加入了一组长上下文相关的参数:

  • --kv_cache_dtype fp8 — 把 KV cache 也降到 FP8,显存占用约减半。这对 128K context 的 DSV4 / Qwen3.5 特别有用。
  • --chunked_prefill_size 1024 — 分块 prefill,降低 prefill 阶段的显存峰值。配合磁盘 MoE('disk':1),即使内存吃紧也能跑长上下文。
  • --page_size — 内部 KV cache 的页大小(类似 vLLM 的 block size,但实现细节不同)。
  • --gpu_mem_ratio — 控制 fastllm 占用显存的比例(类似 vLLM 的 gpu_memory_utilization),典型设 0.85-0.92。
  • --tokens — 限制总 token 数,避免 OOM。

这些参数虽然没达到 vLLM PagedAttention 的成熟度,但说明项目在朝"高并发 + 长上下文"这两个 vLLM 强项靠拢。预计 2026 H2 会有更多这方面的优化。

8.1 fastllm vs KTransformers — 最直接的对手

这两个项目的差异:

  • 设计哲学:KTransformers 仍以 Python + 自有 cpuinfer 算子为主,基于 transformers 库扩展;fastllm 是完全自有 C++ 算子,不依赖 PyTorch(README 第 3 条亮点),启动更轻量。
  • 切层粒度:KTransformers 通常按层数划分 CPU/GPU;fastllm 的 --moe_device "{'cuda':1,'numa':19}" 是按比例分配 MoE 层,且可同时挂 cuda+numa+disk 三种。
  • 张量并行:fastllm 支持奇数张卡 + multicuda + 混合张量并行,KTransformers 多卡支持较弱。
  • FP8 / 量化粒度:fastllm 的动态量化 dtype_config 是 regex-based,粒度比 KTransformers 细。
  • 社区:KTransformers 在英文圈影响力大(尤其 r/LocalLLaMA);fastllm 在中文圈更主流。

8.2 fastllm vs llama.cpp

llama.cpp 也支持 CPU + GPU 混合卸载(-ngl),但:

  • llama.cpp 的 GPU offload 是按层连续切,无法"MoE expert 上 CPU + attention 上 GPU";
  • llama.cpp 的 NUMA 支持较弱(有 --numa 但效果有限);
  • 但 llama.cpp 的 GGUF 量化生态更成熟,Q4_K_M/Q5_K_M 等格式有大量预量化模型。

fastllm V0.1.4.0 起支持读取 GGUF 模型(需 --ori 指定原始模型配置文件夹),实际是把 GGUF 当输入格式,跑自己的算子 — 等于"用 fastllm 的 NUMA + 混合推理跑 GGUF 模型"。这对从 llama.cpp 迁移特别友好。

10 · 1 小时验证 checklist

给读者一个完整 0 → DeepSeek 671B 的命令清单。假设有一台 4090 + 256 GB RAM(可以装 INT4 671B 但不能装 FP8 原版):

# Step 0: 卸载冲突 conda 环境(可选)
conda deactivate

# Step 1: 安装(linux + nvidia)
pip install ftllm -U
ftllm -v        # 验证安装,应该输出版本号

# Step 2: 跑一个 0.6B 小模型验证
ftllm run Qwen/Qwen3-0.6B
# <input: hello>
# <output: ...>
# 退出 Ctrl-D

# Step 3: 跑 Qwen3-8B INT4 验证显卡正常
ftllm server Qwen/Qwen3-8B --dtype int4 --port 8080
# 另开 terminal: curl http://localhost:8080/v1/chat/completions ...

# Step 4: 下载 DeepSeek-R1 INT4 (~340 GB)
ftllm download fastllm/DeepSeek-R1-INT4 --cache_dir /mnt/

# Step 5: 关闭 NUMA balancing 并设环境变量
sudo sysctl -w kernel.numa_balancing=0
export FASTLLM_USE_NUMA=ON
export FASTLLM_NUMA_THREADS=27

# Step 6: 启动 671B 满血 API server
ftllm server fastllm/DeepSeek-R1-INT4 \
  --device cuda --moe_device numa \
  --port 8080 -t 1

# Step 7: 验证(应该看到 ~30 tps)
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"DeepSeek-R1","messages":[{"role":"user","content":"什么是 NUMA?"}],"stream":true}'

# Step 8 (可选): 上 webui
ftllm webui fastllm/DeepSeek-R1-INT4 \
  --device cuda --moe_device numa

如果第 6 步报"内存不足",换 --moe_device "{'cuda':1,'numa':8,'disk':1}" 把 1/10 MoE 卸到 SSD。如果 4090 报 OOM(显存),用 --gpu_mem_ratio 0.85 留点显存给 KV cache。

11 · 局限 / 诚实分析

11.1 不是高并发 serving 首选 — fastllm 没有 PagedAttention / RadixAttention / continuous batching 这类专门的 KV cache 管理技术,所以多并发 60+ tps 已经是 INT4 671B 在双路 9004 上的天花板。对于"我要给 1000 个用户同时提供 8B 模型的低延迟服务"这种 vLLM/SGLang 的强项场景,fastllm 不是最优解。
11.2 中文文档为主,英文文档落后 — README_EN 比 README 简短得多(218 行 vs 454 行),很多关键参数(--moe_device 字典语法、FASTLLM_NUMA_THREADS、动态量化)只在中文文档详细。英文社区(r/LocalLLaMA / X / discord)讨论量也远小于 KTransformers。
11.3 没有 frontier model card 直接引用 — 这是和本系列 #19 MCP-Atlas / #28 BFCL 的关键差异。MCP-Atlas 被 Claude Opus 4.7 system card 引用(77.3%),BFCL 被 GPT-5/5.5 system card 引用 — fastllm 作为推理框架不是评测,自然不会被 model card 引用,但它也没在 frontier lab 的"训练 infra 致谢"里出现过(如 DeepSeek/Qwen 团队的 paper acknowledgements)。它是社区项目,不是 frontier lab 的 infra。
11.4 量化质量盲区 — fastllm 的 int4 / int4g / fp8 量化与 vLLM/SGLang 用的 AWQ / GPTQ / Bitsandbytes 等没有直接的 perplexity / MMLU degradation 横向对比。README 提到"减少专家数可以提高推理速度,但可能降低推理准确度",这条警示是诚实的,但具体到不同量化方式的精度损失,目前在公开文档里找不到系统评测。未在公开来源精确确认
11.5 没有 PagedAttention 等先进 KV cache 管理 — V0.1.6.0 加入了 --page_size / --gpu_mem_ratio / --kv_cache_dtype 等参数,说明项目在朝这个方向走,但目前不是 vLLM 同级的 paged 实现。长上下文(128k+)的 prefill 显存峰值是个已知问题,V0.1.6.2 才"限制 DSV4 稀疏预填充临时显存"。
11.6 Mac 不支持 — README 的安装说明只有 Linux + Nvidia / Linux + AMD / Windows + Nvidia 三种,没有 macOS 路径。这意味着用 M-series Mac 的本地开发者只能用 llama.cpp/MLC-LLM/Ollama,fastllm 不是选项。
11.7 测试矩阵口径含糊 — README 写"任意 ≥10 GB 显卡跑满血 671B",但具体测了哪几张卡、显存使用细节(KV cache 多大、batch=1/4/8 各是多少 tps),官方未给完整 benchmark 表。社区报告很多但散落在 issues。未在公开来源精确确认
11.8 V0.1.6.x 节奏快,可能不稳定 — 5 月单月内 V0.1.5.x → V0.1.6.4 五个小版本,DSV4 / 多模态 / NVFP4 / disk MoE 等大改密集发布,生产环境用建议锁定具体小版本(pip install ftllm==0.1.6.4)。
11.9 PIP wheel 大小限制带来的 ROCm/CUDA 依赖外置 — README 明确说"由于目前 PyPI 限制库大小,安装包中不含 ROCM 依赖",必须手动 amdgpu-install --usecase=hiplibsdk,rocm,dkms。CUDA 12+ 也建议手动装。这对 docker 镜像构建友好(可以共享 base image),但对裸机首装是一个隐藏门槛。
11.10 没有 distributed 多机 EP/PD — DeepSeek 的 671B 在 frontier 部署中,SGLang + Mooncake 提出了 EP (Expert Parallelism) + PD-disagg(Prefill-Decode 分离)架构,可以在 16-64 张 H100 / B200 上实现极高吞吐。fastllm 是单机定位 — multicuda 是单机内 TP,不是跨机分布。要做云级 671B serving,SGLang 仍是首选。这是定位差异,不是缺陷
11.11 GGUF 支持还在补 — V0.1.4.0 起加入,V0.1.4.3/4.6/6.1 持续修 q8_0 / Q2_K_R4 等类型;但目前 GGUF 支持仍不如 llama.cpp 完整(很多 i-quants 还未支持)。把 fastllm 当 "GGUF 高速跑路工具"目前不如直接用 ik_llama.cpp 稳。
11.12 Windows 上的 int4 乱码 + dll 依赖 — FAQ 列出多个 Windows 特有问题:MSVC 优化等级 /Ob2/Ob3 会让 int4 输出乱码(改 /Ob1 解决);GPU 编译后启动报 FileNotFoundError 需要手动拷贝 cudart/cublas 的 dll 到 fastllm_tools 目录。Windows 用户首装很可能踩坑,建议优先 WSL。

结语 · 一句话

fastllm 是一个 "中文社区版 KTransformers + ik_llama.cpp 杂交体" — 它不试图取代 vLLM/SGLang 在高并发 serving 的位置,但在"单机 671B + 老 GPU + NUMA + 国产卡"这条赛道上几乎独占。当你只有一张 4090 / A100 + 一台双路 EPYC 服务器,想在本地跑满血 DeepSeek R1 671B,它就是答案。当你的 K80/P100/MI50 已经被 vLLM 抛弃但想物尽其用,它也是答案。

本系列 #19 是训练侧的 RL 框架横评,本篇 #25 是推理侧的对位 — fastllm 在 RL post-training 完成后的"本地部署 + 评测 + agent serving"这一段补齐了图。结合 #27 (≤40B MCP) + #28 (BFCL) + #36 (Mobile),它是中国研究者最务实的推理 infra选项之一。
