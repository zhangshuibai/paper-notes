速读卡片 (TL;DR)

一句话: MCP-Flow 是 TikTok × SJTU × ZJU × UIUC 合作的 ACL 2026 Main 论文,首次把"从公开 MCP marketplace 自动抓真实 server 并合成训练数据"这件事推到 1,166 server / 11,536 tool 的规模(是 TOUCAN 495 server 的 2.4×,2K tool 的 5.8×),用 Playwright web-agent 跨 6 个 marketplace(Smithery / Glama / MCP.so / MCPHub / PipeDream / PulseMCP)爬取 → 本地通过 npm/uvx/SSE 部署 → GPT-4o 出 few-shot 指令 + slot-fill + WizardLM 进化 → DeepSeek-V3 多重过滤,得到 68,733 instruction-function-call pair + 6,439 trajectory。SFT 后的 Qwen3-0.6B / 4B / Llama3.1-8B 在自定 seen/unseen-tool/unseen-server 三 split 上 系统性超过 GPT-4o / Claude-4-Sonnet(unseen-server AST 65.2 vs Claude 48.4);BFCL V3 上 MCP-Flow-Qwen3-4B 单 turn 85.58 / Live 78.09,与 ToolACE-2-8B 持平;GAIA 上把 Qwen3-4B 成功率从 10.68 翻倍到 21.36(+100%)。但只有 6.4K trajectory,比 TOUCAN 的 1.5M 少 235× —— 这是"更多 server / 更少 traj"的方向性补充而不是替换。代码 + 数据已开源(GitHub + HF),但仓库无 LICENSE 文件、HF 也未声明 license(2026-05 现状),下游商用需自行联系作者确认。

§1 Motivation — 为什么 TOUCAN 495 不够,还要再做 1,166

1.1 论文原话:三条"立项痛点"

论文 §1 + §2 + Table 1 列出当前 MCP 训练数据的三大缺口,直接对应 MCP-Flow 的三个 contribution:

"existing MCP studies have yet to undertake the essential step towards data construction. They exhibit several notable limitations: (1) most of them rely on only a small number of servers (≤20) and tools; (2) they predominantly depend on labor-intensive human data collection; and (3) no existing MCP frameworks offer training support, thereby failing to translate benchmark results into tangible improvements of LLMs." — §1

1.2 MCP-Flow 给出的"三件 contribution"

论文 §1 末尾原句列出:

  1. Web-agent-driven automated pipeline — 用 Playwright(MCP-compatible web agent)跨 6 marketplace 增量爬,适应新 server 涌入;
  2. Large-scale high-quality dataset — 同时支持 training / evaluation / retrieval augmentation(RAG);
  3. Compact fine-tuned LLM 系列 — Qwen3-0.6B / Qwen3-4B / Llama3.1-8B 三档,LoRA SFT。

关键差异化点是 "web-agent 而非 hand-written crawler" —— 论文反复强调 Playwright agent 接 high-level instruction,跨 marketplace 只需小幅 prompt 调整,而 TOUCAN 是 hand-written crawler 直接读 Smithery API。结果就是 MCP-Flow 能"轻量适配 6 个不同 marketplace",TOUCAN 只接 Smithery(495 全部来自 Smithery)。

2.2 Playwright web-agent 爬取流程

论文 §3.1 把流程描述得相当具体:

  1. Playwright MCP 接入:Playwright 本身就是 MCP-compatible 的 web agent,LLM 可以直接以"高层指令"驱动它在浏览器里点链接;
  2. Marketplace 自适应导航:agent 跟着"人定义的 workflow"(找到 server 列表页 → 翻页 → 每个 server 点进去 → 抓 JSON 配置)走;不同 marketplace 之间这个 workflow 只需要换 prompt,不需要重写 HTML parser;
  3. Snapshot-based extraction:抓的是 page snapshot 上的 JSON 配置,而不是 HTML 解析,这是它和传统 scrapy crawler 的关键区别;
  4. 增量更新:论文反复强调 "大规模 crawling + pre-process 后,后续只跑增量",这对 MCP 生态每周冒出新 server的现实非常关键。

2.3 去重逻辑

论文给了一条有意思的 heuristic:"用 tool description 列表的完全匹配判等",而不是 server 名 / provider —— 因为同一个 server(比如 context7)会在每个 marketplace 都出现一遍。这条规则的潜在风险是:同一份开源 server 被 fork 后 tool 名一样但描述微改,会被当成两个,导致 server 数虚高;论文未给具体的 collision 统计。

2.4 本地部署:stdio vs SSE

抓到 JSON 配置后,MCP-Flow 用本地 MCP client把 server 都拉起来 —— 这是和 TOUCAN 同一思路(real server,不是 schema 仿真):

  • stdio 类:走 npm(npx -y @smithery/cli@latest run ...)或 uvx 启动;
  • SSE 类:直接连远程 Server-Sent Events URL;
  • 实现基于开源项目 dolphin-mcp;
  • 排除标准:需 API key / 需专有软件的 server 直接弃用。这意味着 1,166 servers 里面"已经过滤掉了大量真实生产场景"(Notion / Slack / Salesforce 等都要 OAuth)。

2.5 实际生效的 server 子集

关键诚实点:论文 §3.3 末尾原话 — "356 servers producing valid responses and trajectories"。也就是说虽然名义上 1,166 server / 11,536 tool,但实际能跑通 trajectory(产生有效 tool response)的只有 356 个 server。剩下 810 个 server(70%)有 tool 列表 + instruction + function call,但没有真实 tool response,因此也没有 trajectory。这解释了为什么 trajectory 数(6,439)远小于 instruction-function-call pair 数(68,733)。

作为对比,TOUCAN 的 495 server 都是全部能跑通的(因为 TOUCAN 把 trajectory 跑通当作 server 筛选的硬条件),所以这两个"server 数"口径不同 —— TOUCAN 是"traj-callable server",MCP-Flow 是"config-discoverable server"。

§3 合成 pipeline — few-shot + slot-fill + WizardLM 进化 + 三重过滤

论文 §3.2 + §3.3 描述了一条"教师-生成 / 规则-填槽 / 进化-加难 / 多层-过滤" 的合成链。整条链是"tool-first"(从工具出发倒推指令),不是"task-first"(从用户任务出发挑工具)—— 这是 MCP-Flow 和 TOUCAN 在哲学上的另一个关键分歧。

3.1 Stage 1: Few-shot Instruction Generation

给定 tool 的 schema + 描述 + 5 个人写示例,让GPT-4o(generator,论文未试更便宜的)对每个 tool 生成 5 条 distinct instruction。注意:

  • 5 条 × 11,536 tool = 理论 57,680 条原始 instruction;
  • "all of which require the use of the target tool" —— 强制 instruction 必须能解出 ground-truth tool;
  • 这就是 ground-truth label 的来源 —— tool 标签不靠人工标,而是因为"是它生成的"自带标签。

3.2 Stage 2: Slot-Fill Revision

生成的 instruction 可能漏掉某些 required parameter(比如 GenerateImageUrl 要求 Prompt 字符串,但 instruction 只说"画一幅画"没说具体内容)。Slot-fill 做两件事:

  1. 缺槽补值:把所有 required param 当 slot,如果 instruction 没指定就自动生成一个 valid value;
  2. 正则修补:Slot-fill 后还可能残留 placeholder({user_email} 之类),用 specialized regex rules(§D.3)检测后从"pre-collected candidates"(本地预备的真实值池)替换。

3.3 Stage 3: WizardLM Evolution

这是 MCP-Flow 在 SFT 数据上能拉开 ToolACE 的关键步骤 —— 复用 WizardLM (Xu et al., 2023) 的 instruction evolution:

  • 随机选 evolution 方向:"concretization"(更具体)或 "reasoning"(加推理步骤);
  • Evolution depth = 2(论文显式参数);
  • Figure 3 给的例子很有说服力:
    • Initial: "Produce a unique image portraying a serene countryside scene by providing a concise text prompt."
    • Revised(slot-fill 后): "...by providing this text prompt."
    • Evolved(WizardLM 后): "Craft a singular visual depiction illustrating a serene rural landscape defined by untouched emerald-green meadows, a vibrant cerulean sky, and a picturesque, weathered timber-built cottage, ensuring the inclusion of subtle wildlife elements such as birds or deer seamlessly integrated into the scene."

3.4 Stage 4: Function Call Generation

把 evolved instruction + tool input schema 给 GPT-4o,让它输出 formatted function call。论文原话:"This generation is straightforward for strong models such as GPT-4o, as it involves no interference factors." —— 因为没有候选 tool 干扰,只是把自然语言映射到 schema。

3.5 Stage 5: Tool Response Collection → Full Trajectory

这是产生 trajectory 的部分:

  1. 把 function call 真发给本地起的 MCP server;
  2. 收到 tool response(真实的 server output);
  3. 再让 assistant LLM(论文未明示哪一个,推测 GPT-4o)总结 tool response → final response;
  4. 整条 = (instruction, function call, tool response, final response) 即 trajectory。

关键:这一步对 1,166 - 356 = 810 个 server 失败 —— deployment / connectivity issue 导致没拿到有效 response。这就是为什么 trajectory(6,439)远少于 function-call pair(68,733)。

3.6 三重 Filtration(§3.3)

过滤层方法阈值作用
1. Embedding Similarity 计算 instruction ↔ tool description 的 embedding cos sim > 0.8 丢掉 避免 instruction 是"几乎照抄 tool 描述",让 tool 选择 trivial
2. Tool Invocation 给 GPT-4o + DeepSeek-V3 看 instruction + (ground-truth tool + 2 个随机干扰 tool),要求它们选对 两个 model 都答错就丢 双模型校验 ground-truth label 可靠性
3. Quality Score DeepSeek-V3 作 LLM judge,给 instruction 和 function call 打 1-10 分 < 6/10 丢掉 过滤低质量样本。论文复用 LiveMCPBench 的发现:DeepSeek-V3 和人类标注吻合度高

Generator 和 Judge 分离:这是合成数据的常见 best practice —— 论文显式说"DeepSeek-V3 is a SOTA reasoning model different from GPT-4o we use for generation",避免 self-judgement bias。

6.2 vs TOUCAN — "广度 vs 深度"的 head-to-head

规模:MCP-Flow 1,166 server > TOUCAN 495(2.4× 多)。Tool:11,536 > 2,000(5.8× 多)。但 trajectory:6,439 << 1,527,259(少 237×)。

互补关系:

  • 需要"广 tool / 单轮 function-call SFT" → 用 MCP-Flow(0.6B 也能逼近 GPT-4o);
  • 需要"深 multi-turn agentic SFT" → 用 TOUCAN(1.5M traj 才够喂 32B);
  • 两个数据合并应该 strictly 更好,但谁也没公开做过 union 实验(2026-05)。

Server 口径:注意一个不同 — TOUCAN 的 "495" 是跑通筛选后的可用 server,MCP-Flow 的 "1,166" 是配置发现的 server。如果 apples-to-apples 对比,MCP-Flow "可调通的 356" 反而少于 TOUCAN 的 495。论文叙述时偏向"discovery scale"而非"executable scale",这是宣传角度差异。

哲学差异:TOUCAN 是"task-first"(先生成多样的任务,再决定调哪个 tool),MCP-Flow 是"tool-first"(每个 tool 5 条 instruction,再写 function call)。这导致 MCP-Flow 的 instruction 必然已知正确工具—— 强信号训 function-call,但弱信号训 tool selection from large pool。

6.3 vs Smithery(#24)— registry-to-trainable 的两次穿越

  • Smithery(2026-05): 7,300+ server,是 MCP 生态的"Docker Hub";
  • TOUCAN 从 Smithery 拉 ~2,800 server 过滤到 495;
  • MCP-Flow 从 Smithery(单一最大来源)+ 5 个其他 marketplace 拉到 1,166 server;
  • 两边都承认 Smithery 是上游;MCP-Flow 的多 marketplace 策略是为了避免"Smithery 一家独大风险"。

6.4 vs #27 ≤40B 小模型 MCP landscape

MCP-Flow 完美契合 #27 当时的"SFT 冷启 + 小模型 RL"recipe:

  • MCP-Flow 训出的 Qwen3-0.6B 在 unseen-server AST 58.0 —— 0.6B 级别小模型从零起步的合理 SFT ceiling;
  • 接下来该跑 ART + MCP•RL(#27 推荐的 RL 框架)在真 MCP server 上做在线 RL;
  • 但论文没做 RL 部分,留为 future work。

§8 局限 / 个人 take(诚实批判)

8.1 论文自报的 Limitations

  1. Server 可靠性:某些 server 连不上 / 返回 invalid / 已废弃。这是所有真 server 路线的通病(TOUCAN 也有)。
  2. Synth → real 分布差异:合成 instruction 不能完全捕捉真实用户行为。
  3. Multi-turn 弱:论文原话 "MCP-Flow currently prioritizes single-turn tool-invocation data, which may not fully reflect the demands of multi-turn agentic workflows" —— 自报第三条 limitation,正是 BFCL V3 Multi-Turn 31.38 输 TOUCAN/ToolACE-2 的原因。

8.2 我们的补充批判

(a) "1,166 server" 的真实可用性方差

论文承认只有 356/1,166 = 30.5% server 真能产出 valid trajectory。这意味着剩下 70% 是"有 schema 但跑不通"的 server,本质上和 ToolACE 的合成 schema 没区别 —— 模型在它们上面学到的也只是"schema 拟合"。从这个角度看,MCP-Flow 的"executable real server" 实际数 = 356,反而比 TOUCAN 的 495 少。论文宣传 1,166 算"discovery scale",apples-to-apples 应该报 356。

(b) "11,536 tool" 在 2026 后的 churn 率

Smithery 在 2026-05 已有 7,300+ server(#24),也就是说 MCP-Flow 2025-10 截图的1,166 / 11,536到 2026-05 已经过期 6+ 月。MCP 生态每周新增 ~50-200 server(我们粗估),加上老 server 失效。2026-05 实际可达成率未在公开来源精确确认,但可以预期已经从 30.5% 下滑到 ~15-20%。这是所有真 MCP 数据集的共同保质期问题 —— Web-agent 增量爬取机制是个解药,但需要论文作者持续维护(目前没看到 quarterly update commitment)。

(c) 6.4K trajectory vs TOUCAN 1.5M — 质量 vs 数量

这是结构性差异:MCP-Flow 把工程力量花在"广 server 抓取",TOUCAN 花在"深 trajectory 采集"。具体影响:

  • MCP-Flow 的 6.4K trajectory 不够喂 7B+ 模型的多轮训练 —— Qwen3-4B 已经是 sweet spot;
  • BFCL V3 Multi-Turn 31.38 落后 TOUCAN-7B 22.62 的对比有点 misleading —— TOUCAN-7B 的 22.62 已经低于 base Qwen3-4B 31.38,说明TOUCAN 7B 在 BFCL 实际是退化的(#22 笔记详述);
  • 真正的比较应该是 MCP-Flow-Qwen3-4B vs TOUCAN-32B —— 论文没做这个对比,可能因为 32B 训练 cost 高。

(d) 没在 MCP-Universe / MCP-Atlas / Toolathlon 上报数

这是 2026 真 MCP 数据集的"考公"基础题(见 #21 / #26):

  • MCP-Universe 231 task — MCP-Flow 没报;
  • MCP-Atlas 1,000 task — MCP-Flow 没报;
  • MCPMark / Toolathlon — 没报。

论文只在自定义 split + BFCL V3 + GAIA 上报。这让 MCP-Flow 的 unseen-server AST 65.2(Llama3.1-8B)难以直接对比其他 MCP bench 上的数据。"在自家 split 上 65 vs Claude 48"这种比法,outsider 不容易判断是数据强还是 split 偏。

(e) Frontier model card 引用?未确认

截至 2026-05,没有任何 frontier model(GPT-5/5.5,Claude Opus 4.7,Gemini 2.5 等)的 system card 显式引用 MCP-Flow(我们搜遍了 #38 landscape 整理出的所有 system card 引用,只有 MCP-Atlas / BFCL 两家被 frontier 引用)。这也容易理解 —— ACL 2026 才接收,model card 周期上还没赶上。

(f) License 缺失对下游商用是硬障碍

已在 §7 强调。如果是 commercial fine-tune 落地,必须先和 wwh0411 联系书面确认 license,否则法律上等同 all-rights-reserved。学术使用尚可。

(g) 训练 ckpt 不发布

论文宣传"compact fine-tuned LLM series" 是三件 contribution之一,但实际没把 ckpt 放 HF。这对"小模型 MCP 能力下游可复用" (#27 主题) 的友好度大打折扣 —— TOUCAN 直接放出 27 个 ckpt,MCP-Flow 只给"你自己用 LLaMA-Factory 训"的菜谱。

(h) 偏 "广而浅" 的数据形态 — 是否适合 RL?

MCP-Flow 6.4K trajectory + 单轮 function-call 为主的形态,不直接适合 RL 阶段(RL 要 roll-out 多步)。理想的 recipe 是:

  1. MCP-Flow 68K fn-call pair → SFT 冷启 0.6B/4B 模型;
  2. TOUCAN 119K SFT subset → 继续 SFT 加深 multi-turn;
  3. MCP-Universe 11 真 server / ART 单卡 RL → RL fine-tune;
  4. BFCL V4 + MCP-Universe 评测。

这条"组合菜谱"目前没有公开 paper 跑过(2026-05)—— 留给后续工作。

8.3 综合定位

MCP-Flow 在 2026 MCP 数据栈里的位置:

  • 是什么: "跨 6 marketplace 的 web-agent 自动化爬取 + tool-first 合成的真 MCP server function-call SFT 数据集"
  • 不是什么: 不是 RL training data,不是 deep multi-turn corpus(那是 TOUCAN 的活),不是 benchmark(MCP-Universe / Atlas 的活)
  • 最大贡献: 把"真 MCP server 数据集的广度天花板"从 TOUCAN 的 495 推到 1,166(2.4×),tool 从 2K 推到 11.5K(5.8×)
  • 最大代价: trajectory 数从 1.5M 砍到 6.4K(0.4%),multi-turn 能力上不来
  • 最值得抄的方法论: Playwright web-agent 跨 marketplace 增量爬,而不是手写 crawler — 这是"应对 MCP 生态快速变化"的工程模板
  • 最大遗憾: 训出的小模型 ckpt 不开源 + 没有 LICENSE 文件

给读者的 actionable 建议:

  • 需要做 SFT 的人,2026-05 的最佳实践: MCP-Flow 68K fn-call(广) + TOUCAN 119K SFT subset(深) 合并(总 ~187K)做单 epoch SFT,然后接 EnvTuning(#29)或 ART MCP•RL(#27)做 RL fine-tune;
  • 下游商用前必须解决 license — 联系 12321254@zju.edu.cn 拿明确书面许可;
  • 不要直接信赖 "1,166 server" 的数字 — 真正能调通的只有 356,且这个数字在 6+ 个月后会继续衰减;
  • 如果你只 care BFCL V4 排行榜,MCP-Flow 不够 — 还得加 ToolMind(#34)的 turn-level mask 或 EnvTuning 的课程 RL。
