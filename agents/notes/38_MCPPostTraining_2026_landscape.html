🎯 TL;DR

  • 2026 H1 是"MCP 真 server 化"的元年 — 2025 Q4 之前几乎所有 tool-use 训练数据都是 LM 模拟的(ToolACE/Hammer/xLAM)或 Python class 模拟(EnvScaler);2026 进来后 TOUCAN(#22)/Smithery(#24)/MCP-Flow(arXiv:2510.24284, 1166 server)/FireFly(arXiv:2605.17558)/MCPHunt(arXiv:2604.27819) 集体让"真 MCP server 抓数据"成为标准做法。
  • BFCL V4 Overall 开源 top-1 仍是 Nanbeige4-3B-Thinking-2511 (51.40%) ≈ 3B 反超所有 ≤32B 同期 — 但 MCP 长 horizon 评测(MCP-Atlas / MCP-Universe / Toolathlon / MCPMark)上 ≤40B 开源模型至今没有任何一款进 top-10。#27 详查
  • SFT-only(TOUCAN 1.5M)和 RL-on-real-MCP(MCP-Universe verl GRPO / ART MCP•RL / MCP-Flow)在 2026 出现明确分叉:SFT 路线靠"大规模合成 + 多教师 distill"压 BFCL,RL 路线靠"少量真 MCP server + 奖励信号 verify"压 MCP-Universe/Mark — 两边互不通用(TOUCAN 没在 MCP-Universe 报,MCP-Universe RL 模型也没在 BFCL 报)。
  • 2026 H1 新进入:7 个新 paper + 5 个新 benchmark + 3 个新 dataset 全数 covered 在 §2-§4。其中最重要的是 2026-05 EnvFactory(arXiv:2605.18703)/ FireFly(arXiv:2605.17558)/ ComplexMCP(arXiv:2605.10787)/ MCPHunt 安全 bench(arXiv:2604.27819)/ InfTool 自演化合成(arXiv:2512.23611)。
  • Trend G: AgentDistill(#37)"training-free + tool-as-knowledge"派 2026 H1 还没有大量后续 — 主流仍是 SFT/RL,但库派的 reuse-cost 优势使其在 production 落地 quicker。

3.4 2026 H1 新进 infra

EnvFactory (2605.18703, 2026-05) — paper 自称 "scalable, robust execution environments + realistic training data that captures implicit human reasoning"。从 abstract 看是 SETA/EnvScaler 直系演进,但加了"robust RL"标语(单 episode 失败不杀整 batch)。repo / 数据 / 模型权重均未在公开来源精确确认,2026-05 才上线。
ATLAS (2603.06713, 2026-03) — "Scaling Agentic Capabilities, Not Context: Efficient Reinforcement Finetuning for Large Toolspaces"。小 LM 在 large toolspace 下的 RL fine-tune,问题假设 eager tool loading saturate context、execution error compound、sparse reward 难学。和 EnvTuning(#29)同问题域但 RL 派。
诚实分析:除了 MCP-Universe / ART / OpenEnv / verifiers 四家,其他通用 RL 框架(NeMo-RL / rLLM / OpenRLHF / SLIME)到 2026-05 仍没有把"真 MCP server 作为 env"内化为一等公民。结果是 — 想训 MCP 模型的研究者只能在 (a) 自己造 server wrapper、(b) 用 ART 单卡跑、(c) 用 MCP-Universe 的 docker_pool、(d) 用 OpenEnv WebSocket 桥接 — 四种方式中挑。

4.2 2026 H1 新进 benchmark

NEW ComplexMCP (arXiv:2605.10787, 2026-05) — "Last mile" 商业软件自动化 bench。关键差异点:tool 不是独立的,而是 atomic、interdependent、有 environmental noise。直击 MCP-Universe/Atlas 在"工具组合 + 噪声鲁棒性"上的盲区。规模 / leaderboard / 数据 license 均未在公开来源精确确认。
NEW MCPHunt (arXiv:2604.27819, 2026-04) — 2026 第一个 MCP 安全 bench。问题:multi-server MCP agent 创建的 information-flow control 问题,benign read/write permission 在多 server 间组合 → cross-boundary credential propagation。HF dataset: mcphunt-benchmark/mcphunt-agent-traces 已上线(235 下载)。是 MCP 走向 production 前 ops/security 团队最关心的第一个 bench。
NEW TOBench (arXiv:2605.16909, 2026-05) — Task-Oriented Omni-Modal Benchmark for Real-World Tool-Using Agents。强调 tool use + computer use + multimodal reasoning 三合一(此前的 bench 三者分割)。MCP-Universe / Atlas 都没有 vision 输入,这是第一个把视觉 + tool 合一的 bench。规模 / leaderboard 未在公开来源精确确认。
NEW MCPBench (arXiv:2504.11094, 2025-04) — 早于 MCP-Universe 的小规模 MCP 评测,主要关心 accuracy + time + token usage。规模较小,主要价值是"第一篇 cite 的 MCP-specific eval paper"。
NEW CUBE (arXiv:2603.15798, 2026-03) — "Standard for Unifying Agent Benchmarks"。这是元 benchmark,定义统一 schema 用以横跨 BFCL/MCP-Universe/Toolathlon 等。如果工业界采用,有可能成为 2026 H2 的"agent eval ONNX"。
NEW EvoMemBench (arXiv:2605.18421, 2026-05) — agent 记忆能力 self-evolving bench。和 BFCL V4 memory_kv/vector/rec_sum 重叠但更长 horizon。
NEW CHI-Bench (arXiv:2605.16679, 2026-05) — "Can AI Agents Automate End-to-End, Long-Horizon, Policy-Rich Healthcare Workflows?"。医疗领域的长 horizon agent eval,domain-specific 但和 MCP-Universe 在"长 horizon + policy compliance"维度上重叠。
NEW Herculean (arXiv:2605.14355, 2026-05) — "Agentic Benchmark for Financial Intelligence"。金融领域 agent bench,延续 MCP-Universe Finance domain 的方向。
NEW TPS-Bench (arXiv:2511.01527, 2025-11) — "Tool Planning & Scheduling Abilities in Compounding Tasks"。直接挑战 ComplexMCP 同问题(tool 依赖、组合调度),早于 ComplexMCP 半年提出。

7 · Trends — 8 条 meta-pattern

Trend A: 真 MCP server vs 合成 env — 谁在赢?

2025 Q4 之前:合成 env 派完胜 — EnvScaler / SETA / AWM 都在合成 env 上 RL 出 SOTA 小模型。2026 Q1-Q2:真 MCP server 派后来居上 — TOUCAN(495 real server)、MCP-Universe(11 real)、MCP-Flow(1,166 real)、FireFly(real + verify-first)。原因有 3 条:

  • 合成 env 顶到了 sim-to-real gap:Qwen3-Thinking-8B 在 EnvScaler 合成 BFCL-MT 41.88,但在 MCP-Universe 真 server 上跑不到 25% — sim-to-real gap 在 ~17 pp。
  • 真 server 数据已经规模化:Smithery(#24)2026-05 已有 7,300+ server,TOUCAN/MCP-Flow 都已从中抓出 1K+ server / 11K+ tool 的训练规模。
  • verify 闭环只有真 server 能给:FireFly(2605.17558)和 MCP-Universe 的 dynamic verifier 都是基于真 server 副作用 — 合成 env 的 verify 必然是 LM 模拟的,信号脏。

预测:2026 H2 主流会变成"真 MCP cold-start + 合成 env 长 horizon RL"两阶段 — 合成 env 不死,但降级为"扩展真 server 没覆盖的 long-tail domain"。

Trend B: SFT vs RL — 谁更可 scale?

这是 2026 最有争议的问题。三组观察:

路线代表规模BFCL V3/V4MCP-Universe
纯 SFT(无 RL)TOUCAN-32B(#22)1.5M traj · 真 MCPV3 Overall 70.45(险胜 GPT-4.5)Pareto 推进但~25-30%(无绝对数)

结论:2026 H1 没有任何一条路单独 dominant。但SFT 路线在"数据规模"维度上有压倒优势(TOUCAN 1.5M >> EnvScaler 9K >> MCP-Universe 231),而 RL 路线在"信号干净度"维度上有优势(可验证 vs LM-judge)。预测 2026 H2 的主流 recipe 是 SFT 冷启 + RL 微调(类似 ReTool/Search-R1/DeepSeek-R1 的两阶段)。

Trend C: Cold-start data + RL second stage 是否已成标准?

从 #22(纯 SFT)、#23(纯 RL)、#34(SFT + turn-mask filter)、#29(SFT + 课程)、#27 ART MCP•RL(SFT 冷启可选 + RL)、#11 ReTool(SFT 冷启 + RL)、#10 Search-R1(纯 RL)看 — 已成主流但还没绝对一致。具体到 MCP:

  • 多数研究者默认两阶段:ToolMind 的 4 (graph-walk SFT) + 4 (binary filter) 是 SFT-only 但已含两层信号;ART MCP•RL 默认"先 SFT 一份再 RL"; MCP-Universe verl 训练栈本身就支持两阶段。
  • 纯 RL 派坚持单阶段:Snowflake AWM、SETA、EnvScaler、Agent-World 都是从 Qwen3-Thinking 直接 RL,无 SFT 中间层。
  • 纯 SFT 派也坚持:TOUCAN 论文标题就把"only SFT, no RL"写进 contribution。

结论:Cold-start SFT + RL 是事实标准但不是唯一;选哪条取决于数据来源 — 真 MCP 数据多就 SFT-only,真 MCP 数据少就 RL 弥补。

Trend E: MCP-specific spec tokens 进 tokenizer 化

这是 user 最近发现的一条线 — tokenizer 层把 MCP 协议元素当 special token 处理:

  • Qwen3.6(预期 2026 Q3):tokenizer 据报已加入 <tool_call> / <|tool_call|> 作为 protocol token。
  • Gemma 4(预期 2026 Q3):tokenizer 据报已加入类似 protocol token。
  • Claude / GPT-5:tokenizer 已经长期内置 tool-call special token,但 Anthropic / OpenAI 不公开 vocab。

意义:"function call"从 prompt template 升格为 tokenizer level primitive,意味着 (a) 数据集需要保留这些 special token 不被替换,(b) post-training 的 multi-turn 模板必须用 native token 而非 [TOOL_CALL] 文本 placeholder,(c) ART MCP•RL / verifiers / MCP-Universe 的训练 framework 需要支持新 token。

"Qwen3.6 / Gemma 4 添加 tool_call protocol token"是 user 的近期发现,具体哪个 minor version、token 字符串精确形式 — 未在公开 release note 精确确认。但这条 trend 与 OpenAI/Anthropic 的 closed practice 一致,业界趋势没有疑问。

Trend G: Mobile / on-device MCP

从 #36 移动 landscape 看,2026 H1 "移动设备上跑 MCP"几乎是空白:

  • Apple Foundation 3B / Gemini Nano 3.25B 模型有 native tool support 但不连真 MCP server(iOS 跨 app 权限模型不允许)。
  • 社区 MLC-LLM / ExecuTorch 已能在 iPhone 跑 ≤4B,但 MCP server 在手机上没 host(server 通常 docker,手机无 docker)。
  • 2026 H1 唯一相关工作:FPHam/kitten_tts_mcp(HF · TTS-as-MCP-tool · 2026-03)— TTS engine 包装成 MCP server,跑在 Windows 而非 iOS/Android。

open opportunity:把 MCP client 跑在 phone(连云端 MCP server)+ 1.5-4B Qwen3 / Apple Foundation,做 personal assistant — 这是 2026 H2 最大的空地之一。

Trend G': Domain-specific MCP bench 涌现(垂直化)

2026 Q1-Q2 一波领域专用 agent bench:Herculean(金融,2605.14355)、CHI-Bench(医疗,2605.16679)、TPS-Bench(scheduling,2511.01527)、TOBench(multimodal,2605.16909)、EvoMemBench(memory,2605.18421)、CTFusion(CTF,2605.11504)、MCPHunt(security,2604.27819)。这些 bench 都不直接用 MCP-Universe / Atlas 的 11-36 server,而是 domain-specific 真 server。意义:

  • 2026 前的 MCP eval 是"general purpose,跑 11-36 个通用 server",2026 后是"vertical-purpose,跑 5-10 个领域专 server"。
  • 这条 trend 对模型训练的影响:cold-start SFT 通用,RL 微调 vertical。比如训一个 ToolACE-Finance / TOUCAN-Healthcare 在某 vertical bench 上专精。
  • 开源还没有任何 vertical-specific MCP 模型,这又是一个 open opportunity。

Trend H: AgentDistill / training-free 派

从 #37 看,"training-free + tool-as-knowledge"派提出的 MCP-Box(把 capability 搬到运行时工具库)在 2026 H1 仍是小众。

  • 主流仍是 SFT/RL 派(TOUCAN/ToolMind/ToolACE/EnvScaler/SETA),把 capability 放进权重。
  • 但库派的 reuse-cost 优势使其在 production 落地 quicker — 商业落地不愿意每个新 MCP server 都重训。
  • 预期 2026 H2 会出现 RAG-over-MCP-Box(检索式调用 + retrieval-augmented MCP tool),把训练成本压到接近 0。

8.1 Recipe A 详细步骤(单 H100,1 周)

# Day 1-2: 数据准备
hf download Agent-Ark/Toucan-1.5M           # 1.5M traj 全量

  --model Qwen/Qwen3-8B-Thinking \
  --dataset ./toucan_100k.jsonl \
  --bf16 --batch 16 --lr 2e-5 \
  --epochs 1 --max_length 16384

git clone https://github.com/OpenPipe/ART
cd ART && pip install -e .
python -m art.mcp_rl \
  --base ./checkpoints/sft \

  --episodes 200 --gpu 1
# 单 H100 / ~12h

# Day 7: eval
bfcl evaluate --model ./checkpoints/rl --test-categories all

10 · Open questions / 局限

10.1 2026 H1 还没有人做的事

  • ≤40B 开源模型在 MCP long-horizon bench 上的 top-10 record — 4 大 bench(Universe/Atlas/Mark/Toolathlon)集体 0 ≤40B 开源 record。这是 NeurIPS 2026 / ICLR 2027 最大 open opportunity。
  • 移动端真 MCP client(#36)— iPhone/Android 上跑 3-7B 模型 + 连云端 MCP server,2026-05 还没有任何 production app。
  • MCP-Box(#37)的可 scale 实现 — AgentDistill 只在 K≤13 tool 时 work,2K+ tool 时退化为 RAG。还没人做 retrieval-augmented MCP-Box。
  • 多模态 MCP tool call — TOBench(2605.16909)是 2026-05 第一个开口,但训练数据稀缺,大多 MCP server 还没视觉输入。
  • MCP-native tokenizer 训练实证 — Qwen3.6 / Gemma 4 加 tool_call token 后,从 scratch retrain 一遍证明 +X pp,还没人做这个对照实验。
  • MCP server 安全测试集 — MCPHunt(2604.27819)开了个头,但只覆盖 cross-boundary credential propagation,prompt injection / privilege escalation / data leak 还是空白。
  • 真 MCP server 上的 RL 框架 scale 验证 — MCP-Universe 用 11 server,如果上到 100 / 1000 server(Smithery 规模),docker_pool 是否还能 scale 没人测。
  • SFT 数据规模天花板 — TOUCAN 1.5M 已是 2026 最大,但还没人测过 10M / 100M SFT 是否还有收益。

10.2 仍然没有的 benchmark 形态

  • 多 user agent + MCP server 同时跑:目前所有 MCP bench 都是单 agent 跑单 server,没有"两个 agent 共用同一 MCP server"的 multi-tenant 测试。
  • 真实持续 deployment 评测:所有 bench 都是 episode-based,没人测 30 天连续运行 MCP agent 的 drift / staleness / state corruption。
  • cross-region / cross-language MCP server:Smithery 主要英文,中文 MCP server 生态在 ACEBench 测但不是 MCP-specific。
  • cost-aware MCP eval:目前所有 bench 都按 accuracy 排,没人在 leaderboard 上同时显示 "accuracy / $token / $minute"。

个人 take

看过 #19-#37 共 20 篇深读 + 2026 H1 新进的 12 个工作后,2026 MCP 后训练的核心矛盾是:

  1. 数据规模(TOUCAN 1.5M)已不是瓶颈,reward 信号干净度才是 — 这就是为什么 FireFly (verify-first)和 MCP-Universe (dynamic verifier) 比 TOUCAN (SFT-only) 更有 long-horizon 潜力。
  2. ≤40B 开源模型在 MCP 长 horizon 上集体缺席,但 BFCL V4 上 3B 反超 32B(Nanbeige4) — 说明 small-model + 高质量数据可行,只是没人愿意花 8×H100 一个月训一个 3B MCP 模型。
  3. MCP-Atlas 是 PR 标杆,MCP-Universe 是技术标杆 — 但两者都没大量被 ≤40B 开源跑过。学术界跑 Universe / Mark / Toolathlon 都不约而同避开 ≤40B 数据 — 这就是 2026 H2 必须填的洞。
  4. 2026 H1 最有可能爆款的工作是 真 MCP server 数据集 + 课程 RL 的两阶段 recipe。MCP-Flow(1,166 server)+ EnvTuning(4-stage 课程)+ ART(Apache 单卡 RL)三件套已经搭好了,差一个愿意把它跑通 + 在 MCP-Universe 上报数字的研究者。
